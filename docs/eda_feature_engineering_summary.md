## 1. 가장 의미있고 중요했던 것 (Tier S — 방법론/인프라)

이 항목들은 "점수를 올린 것"이 아니라 **"제대로 실험할 수 있게 만들어준 것"**이라 가치가 가장 큽니다. 이게 없었다면 나머지 시도들의 결과 자체를 신뢰할 수 없었습니다.

### 1-1. 공식 평가 산식 구현 + 진단 도구 (evaluate.py)
- FICR이 계단식(6%/8% 문턱) 구조라는 걸 파악하고, `error_rate_breakdown`, `bias_diagnosis`, `bias_by_prediction_quantile` 등 진단 함수 구축
- **핵심 발견**: 1-NMAE는 준수(0.87)한데 FICR이 낮은(0.30) 원인이 "8% 초과 시간대가 60%대"라는 걸 수치로 확정 → 이후 모든 의사결정의 나침반이 됨

### 1-2. 3구간 분리(학습/보정/holdout) + calib 기반 의사결정 구조
- holdout은 절대 튜닝에 안 쓴다는 원칙을 처음부터 고정
- 이 원칙 덕분에 나중에 v12에서 "holdout 자체가 부풀려져 있었다"는 걸 자체적으로 발견할 수 있었음

### 1-3. backtest.py — 다중 시기(5윈도우) 검증 인프라
- **이번 프로젝트에서 가장 중요한 전환점.** v8~v12까지 반복된 "홀드아웃에서 좋아 보이는데 실제론 아닌" 패턴의 근본 원인(단일 구간 우연)을 밝혀냄
- 이후 모든 실험(v14~v22)이 이 인프라 없이는 방향을 못 잡았을 것

### 1-4. VESTAS SCADA 이상치 규명 (diagnose_vestas.py)
- 음수값 13~24%, 정격의 만 배 이상 극단치가 소수 존재한다는 걸 정량적으로 확인
- 원인을 몰랐다면 "VESTAS는 원래 이렇게 노이즈가 심한가보다"로 오해하고 그룹1·2(전체 용량의 2/3)를 영영 포기했을 것

### 1-5. 팀원 EDA — 벡터 상쇄 문제 / SCADA `.sum()` 버그 / 시간 컨벤션
- 우리 코드에 실제로 존재하던 방법론적 오류 3가지를 실증으로 짚어줌 (r=0.9998 등 정량 검증 포함)
- 성능 개선으로 직결되진 않았지만(v21, v22), **"틀린 걸 쓰고 있었다"는 걸 몰랐다면 이후 어떤 시도도 신뢰할 수 없었을 것**이라는 점에서 방법론적 가치가 큼

### 1-6. MAE objective 정렬 + 시드 앙상블 (v11)
- 유일하게 "선택이 아니라 원리에 기반한 결정" + "무조건 평균내는 안전한 기법"으로 실제 리더보드에서 검증된 개선
- 지금까지 실제 최고 기록(0.6146)의 근거

## 2. 중요했지만 임팩트가 제한적이었던 것 (Tier A)

시도 자체는 타당했고 로컬(backtest) 지표는 개선됐지만, 실제 리더보드에서는 확인되지 않았거나 미미했던 것들.

| 시도 | 로컬 신호 | 실전 결과 | 평가 |
|---|---|---|---|
| v17: 공기밀도/윈드시어/인접시간대 | backtest 최초로 정체구간 돌파 | 0.6141 (v11과 동급) | 방향은 맞았을 가능성(실제로 850hPa 상층풍·wind_shear·lag/lead가 v22 feature importance 상위권에 다수 포진) |
| v18: 격자간 분산(std) | backtest 전체 최고(raw 0.5968) | 0.6134 (동급) | 마찬가지로 importance 상위권에 듦 |
| v20: log1p 타겟 변환 | calib 기준 backtest 최고(0.5989) | **0.6072 (최저권)** | "raw 자체는 나빠지고 보정이 상쇄" 구조라 불안정 — 로컬 신호를 과신하면 안 된다는 반면교사 |
| 모델 블렌딩(XGBoost, v15) | 유의미한 차이 없음 | 미제출 | 시드 앙상블만큼의 다양성 효과는 없었음. 다만 구조(modeling.py)는 계속 유용하게 재사용됨 |

**공통 교훈**: v17·v18의 "새로 만든 피처"들이 v22 feature importance 상위 10위 안에 실제로 여러 개 포진해 있음(`gfs_wind_shear`, `ldaps_ws10_gridfirst_max`, `ldaps_ws10_speed_lead1/lag1`). 즉 **모델이 이 피처들을 실제로 유용하게 쓰고는 있는데, 그게 최종 Score를 리더보드 기준으로 유의미하게 밀어올리진 못했다**는 것 — "쓸모없는 피처"와 "쓰긴 하는데 결정적이지 않은 피처"는 다르다는 걸 보여줌.

## 3. 시도했지만 효과가 없었던 것 (Tier B — 기각)

| 시도 | 가설 | 기각 사유 |
|---|---|---|
| IDW 거리가중평균 (v8) | 가까운 격자에 더 큰 가중치 | 최근접 3격자가 이미 1.5km 이내로 촘촘해서 단순평균과 차이 없음 |
| 컷아웃/고풍속 플래그 (v9) | 극단적 고풍속 구간 별도 표시 | 예보 풍속과 SCADA 실측 풍속의 절대 스케일이 안 맞아 임계값이 무의미했음 |
| 하이퍼파라미터 그리드 탐색 (v10) | 최적 조합 탐색 | calib(60일)이 여러 후보 중 고르기엔 too small → 과적합. **이후 "선택하지 않고 확정 규칙으로" 원칙의 계기가 됨** |
| 파워커브 직접 연결 (v14) | 예보풍속→발전량 매핑을 명시적 피처로 | 트리 모델이 원본 풍속 피처로 이미 같은 관계를 학습 중 → 중복 정보 |
| SCADA 물리 피처 v16 (버그 있던 버전) | 진짜 물리 파워커브를 2단계로 활용 | 나중에(v22) `.mean()` 집계로 실제값이 1/6로 축소되는 스케일 버그가 있었던 것으로 확인. **버그 수정 후(v22)도 여전히 개선 없었음** — 정보 자체는 트리가 이미 흡수하고 있었을 가능성 |
| 공기밀도×풍속³ 상호작용항 (v19) | 물리식을 명시적 곱으로 제공 | 트리 모델은 분할 과정에서 상호작용을 스스로 학습 가능 — 명시적으로 안 줘도 됨 |
| 그룹별 격자집계 차별화 (v22) | 그룹1은 전체평균, 그�룹2·3은 최근접 | 팀원의 상관분석 근거는 탄탄했으나 실제 backtest 성능은 동일 |
| 최종 제출 raw vs 보정 (v13, v20) | 보정 없이/있이 비교 | 두 방향 다 실전에서 일관된 우위 없음(v13 최저 0.6073, v20 최저 0.6072) → **보정 여부 자체는 큰 지렛대가 아님을 확정** |

**패턴**: Tier B의 상당수가 "트리 모델이 이미 스스로 학습할 수 있는 정보를 명시적으로 다시 제공"하는 시도였음. 이건 이번 프로젝트의 중요한 학습 포인트 — **정형 데이터 + 트리 모델 조합에서는 물리적으로 그럴듯한 상호작용/파생 피처를 손으로 만드는 것의 한계효용이 낮다.**

## 4. 아직 시도하지 않은 것

### 4-1. 피처/전처리 단계에서 남은 후보 (우선순위 낮음 — 이미 소진 판단)
- **터빈 단위 개별 예측 후 그룹 합산**: 지금은 그룹 단위로 뭉쳐서 예측. 12기(그룹1·2)를 개별로 예측하면 노이즈 패턴이 다를 수 있으나, 리팩터링 비용이 크고 VESTAS 데이터 자체 품질 문제가 여전히 걸림
- **레짐 분리 모델**: 평온/강풍 구간을 완전히 별도 모델로 분리 (v9의 "플래그 피처 추가"보다 강한 형태의 시도로, 아직 "별도 모델"까지는 안 해봄)
- **예측값 후처리 스무딩**: 인접 시간대 예측값을 사후에 평균/보정 — 8% 문턱을 겨우 넘는 튀는 값 완화 목적, 한 번도 시도 안 함
- **외부데이터(태풍 이력 등)**: 대회 규정상 재분석자료/사후관측 사용 금지로 자유도가 제한적. 우선순위 최하로 이미 합의됨

### 4-2. 모델 구조 단계 (다음 트랙)
- **LSTM/시계열 신경망**: 매 시간을 독립적으로 취급하는 현재 구조의 근본적 대안. 다만 데이터 규모상 트리 모델을 이기기 쉽지 않을 수 있어 별도 트랙으로 검증 예정
- **트리 모델과 신경망 블렌딩**: 성격이 다른 두 모델의 오차 패턴이 다르다면 앙상블 시너지 가능성

### 4-3. 검증 방법론 단계
- **k-fold 방식으로 하이퍼파라미터 재탐색**: v10이 실패한 건 calib 데이터가 작아서였음. calib을 늘리거나 k-fold를 도입하면 재시도 여지가 있으나, 지금까지 "미세조정 자체의 한계"가 반복 확인된 만큼 우선순위 낮음

## 5. 종합 결론

1. **실제 리더보드는 20개 버전에 걸쳐 0.5937~0.6146 사이에서 정체**되어 있고, 최고 기록은 v11(0.6146)이 여전히 유효함
2. 가장 가치 있었던 산출물은 **"점수를 올린 것"이 아니라 "제대로 검증할 수 있는 인프라(backtest.py)와 진단 도구(evaluate.py, diagnose_*.py)"**였음
3. 트리 모델(LightGBM/XGBoost) + 현재 피처셋의 조합은 **실질적 상한선에 도달한 것으로 판단** — 물리적으로 타당한 새 피처를 계속 추가해도(공기밀도, 파워커브, 상호작용항, SCADA 물리모델) 트리가 이미 학습 가능한 정보와 중복되는 경우가 대부분이었음
4. 팀원 EDA로 실제 코드 버그 2건(벡터 상쇄, SCADA 스케일)을 발견하고 수정했다는 것 자체가 **모델 성능과 무관하게 프로젝트의 신뢰도를 높인 성과**
5. 다음 단계로 **모델 구조 자체를 바꾸는 시도(LSTM 등)**가 합리적 — 단, 검증은 반드시 backtest.py 다중윈도우 기준으로 먼저 하고, 실제 리더보드 제출은 확실한 신호가 있을 때만 아껴서 사용
