발견 사항:
- **전체 시간의 약 39~46%가 아예 평가에 안 들어감**(10% 적격기준 미달) — 평가페이지에서 읽은 PLAN.md의 "약 54~61%가 기준을 넘는다"는 수치와 대략 일치(기간/반올림 차이 감안). 이게 모델링의 지배적 제약조건임을 확인, 사소한 예외가 아님.
- **하드 클리핑/포화 상한 없음.** 설비용량 대비 `[90,95)%, [95,98)%, [98,100)%, [100,110)%` 구간의 밀도를 확인 — 세 그룹 모두 100%로 갈수록 밀도가 단조 감소, 어느 임계값에서도 spike/plateau 없음. `[가설]`: 정산 계량기가 실제 순출력을 그대로 반영하거나, 이 데이터셋에서 curtailment(출력제한)가 드물거나 하드한 MW 상한이 아닌 소프트한 형태로 나타남을 시사 — **미확인**, 제공 데이터에 직접 대조할 curtailment 플래그가 없음.
- **그룹3은 38시간(0.22%)에서 설비용량을 초과**, 최대 CF 1.006. 작고 그럴듯한 수준(단기 초과출력/계량기 오차범위는 풍력 데이터에서 흔함)이지만, 제출 시 `clip(0, capacity)` 처리할지 평가 스크립트가 클리핑을 안 한다면 1을 살짝 넘는 값을 허용할지는 결정이 필요.
- **계절 패턴**(`target_seasonal_hourly.png`, 월별 평균CF): 겨울철(11월~2월)이 여름철(6~8월)보다 세 그룹 모두 뚜렷하게 평균CF가 높음 — 한국 동해안 겨울 계절풍 기후와 일치. `[인과 메커니즘은 가설, 데이터 패턴 자체는 검증됨]`.
- **일중 패턴**: 계절 신호보다 훨씬 평평하지만 완전히 평평하진 않음 — 모든 그룹의 시간대별 평균CF 곡선에서 오후에 완만한 하락이 보임. 계절 대비 효과 크기는 작음. 순환 시간 피처는 넣을 가치 있지만 원시 풍속만큼 비중을 크게 기대하진 말 것.
- **그룹3은 그룹1/2보다 저CF 꼬리가 더 두꺼움**(zero-frac 더 높음, 10% 미만 비율 더 높음) — 라벨 이력이 짧은 것(2023년 시작)을 감안해도, 이는 실제 사이트 수준의 차이(UNISON U136 사이트 특성)이지 단순히 데이터 적음의 부산물이 아님.

## 4. 타겟 vs 기상 상관관계 `[검증됨]`

방법: 각 기상 소스를 `forecast_kst_dtm`당 1행으로 집계할 때 **그리드 전체 단순평균**을 사용(농장으로의 공간 축소는 아직 안 함 — 1차 패스일 뿐, 최종 피처 아님), `u/v → 속도/방향` 계산, `kst_dtm == forecast_kst_dtm`으로 `train_labels`와 조인(이 조인 키가 맞는 이유는 §6 참고). 전체 표는 `docs/eda_correlation_table.csv`. **전체 시간**과 **적격 시간만**(실적 ≥ 설비용량10%, 즉 실제로 채점되는 부분집합) 둘 다 보고 — masked objective에선 이 둘이 의미있게 다를 수 있어서.

|Pearson| 기준 랭킹, 전체시간, 그룹1(그룹2/3도 같은 순서로 유사하게 추종):

| 피처 | Pearson(전체) | Pearson(적격만) |
|---|---:|---:|
| LDAPS 10m 풍속 | 0.727 | 0.582 |
| LDAPS 10m 풍속(Spearman) | 0.751 | — |
| LDAPS↔GFS 10m 속도 불일치 | 0.600 | 0.438 |
| GFS 100m 풍속 | 0.601 | 0.452 |
| GFS 10m 풍속 | 0.585 | 0.456 |
| GFS 지상 돌풍(gust) | 0.569 | 0.430 |
| LDAPS 10m 풍속³ | 0.573 | — |
| LDAPS 10m 풍향 | 0.422 | 0.292 |
| LDAPS 2m 기온 | -0.315 | -0.281 |
| GFS 2m 기온 | -0.316 | -0.277 |
| LDAPS 지표기압 | -0.218 | -0.141 |
| GFS 전운량 | -0.180 | -0.190 |
| LDAPS 2m 상대습도 | -0.076 | -0.026 |

발견 사항:
- **풍속이 압도적**, 예상대로. 하지만 LDAPS 10m 풍속에서 **Spearman(0.751) > Pearson(0.727)** — 파워커브가 선형이 아니라 단조증가하되 비선형(시그모이드/큐빅 후 포화)이라는 것과 일치. 원시 `u/v`, `speed`, `speed³` 전부 GBM에 넣는 게 낫다 — 모델이 알맞은 비선형을 스스로 찾을 것. 다만 `speed³` 단독은 원시 `speed`보다 나은 게 아님(0.573 vs 0.727) — 순진한 물리적 큐빅 피처가 지름길은 아님, 과대평가하지 말 것(물리 파워커브 베이스라인 모델 전용으로는 여전히 괜찮음).
- **적격시간만 보면 상관이 뚜렷하게 떨어짐**(예: LDAPS 풍속 0.727→0.582). 예상된 결과 — CF≥10%로 제한하면 원시 상관의 대부분을 차지하던 "잔잔함 vs 아님" 신호가 빠지고, 더 어려운 "이미 바람이 부는 상황에서 얼마나 부는가" 신호만 남음. PLAN.md §1.3의 지적(적격시간만으로 최적화하는 모델은 마스크 없는 회귀와는 의미있게 다른, 더 어려운 문제를 푸는 것)을 직접 뒷받침.
- **온도가 실질적인 2차 신호**(양 소스 모두, 양 마스크 모두 ~-0.3) — 공기밀도의 프록시일 가능성(추울수록 공기 밀도 높아 같은 풍속에서도 출력 더 큼)과/또는 계절 프록시. `[가설]` — 밀도 메커니즘 자체는 계절 교란변수와 분리 안 됨.
- **LDAPS↔GFS 불일치(0.60)가 GFS 자체 풍속(0.585~0.601)만큼 예측력 있음** — PLAN.md §1.4의 "암묵적 앙상블/불일치 신호" 아이디어를 뒷받침할 피처로 유지할 가치 있음. 이번 1차 패스로 이론이 아니라 실증적으로 지지됨.
- **풍향 상관(0.42/0.29)이 "별로 안 중요할 것" 치고는 의외로 높음** — `[가설]`: 사이트 지형/wake(후류) 노출이 방향에 의존적이기 때문일 가능성이 높음(어떤 방향은 개활 능선을 마주하고, 어떤 방향은 상류 터빈열의 wake 그늘에 들어감), 방향 자체가 직접 출력을 좌우하는 게 아니라. `sin/cos(방향)` 피처로 유지해서 모델이 wake 섹터 효과를 스스로 찾게 하는 게 좋음 — 메커니즘을 증명하기엔 부족함.
- **구름량/습도/기압은 약하지만 0은 아님** — 바람과 상관있는 전선/폭풍 조건의 부차적 프록시일 가능성. 1차 피처셋에선 낮은 우선순위.

그림: `power_curve_scatter.png` — 그룹별 LDAPS 10m 풍속 vs 설비이용률 산점도. 육안으로 파워커브 형태(급경사 램프 이후, 깔끔한 포화선이 아니라 ~8~15m/s 구간에서 넓고 분산이 큰 "고원" 밴드) — 이 고원의 *폭*(cut-in 이후 특정 풍속에서의 높은 분산) 자체가 정보: 그리드평균 해상도에서는 풍속만으로 설명 안 되는 분산이 많이 남고, 그게 바로 다른 피처들(온도/밀도, 방향/wake, LDAPS-GFS 스프레드, 난류 프록시)이 채울 수 있는 여지.

## 6. 타임스탬프 정렬 — label ↔ weather ↔ SCADA `[검증됨]`

PLAN.md §1.4(리크 없는 조인)와 §4.1(아키텍처가 이걸 제대로 맞추는 데 의존)에서 지적한 가장 큰 구조적 리스크를 직접 해소:

- **`kst_dtm`(라벨) = 시간 끝(end-of-hour) 컨벤션, SCADA로 대조 검증.** 10분 단위 SCADA 파워를 `resample("1h", label="right", closed="right")`(데이터 사전의 "실제 발전량 집계 구간의 종료 시각"과 일치하는, 각 `kst_dtm`에서 끝나는 시간)로 시간당 집계하고, 그룹1 터빈 6기 합산값을 `kpx_group_1`과 비교: **Pearson r = 0.9998**, 비율 중앙값 0.986. `label="left"`나 ±1시간 시프트는 전부 명확히 더 나쁜 정렬(r ≤ 0.97) — `right`/시프트 0이 명백히 맞음.
- **SCADA의 `*_power_kw10m` 컬럼명은 오해의 소지가 있음 — 이름과 달리 순간 전력(kW)이 아니라 10분 구간 에너지(kWh)임.** 시간당 6개 10분 판독값에 대해 `.mean()`과 `.sum()`을 둘 다 그룹 라벨과 대조해서 발견: `.mean()`(전력으로 취급)은 실제 라벨의 약 1/6 값을 줌(비율≈5.92, 즉 시간당 판독 횟수 6과 정확히 일치하는 약 6배 오차); `.sum()`(10분 에너지로 취급)은 비율≈0.986, 사실상 1:1. **이건 피처엔지니어링의 실질적 함정** — SCADA 기반 파워커브나 대조 피처를 만들 때는 반드시 시간당 `.sum()`을 써야지 `.mean()`을 쓰면 조용히 ~6배 축소된 값이 나옴.
- **`forecast_kst_dtm == kst_dtm`이 맞는 조인 키임(시프트 불필요).** 데이터 사전에 직접 명시되어 있고("발전량을 예측해야 하는 시각과 대응됩니다"), 실증적으로도 확인됨: `kpx_group_1`을 LDAPS 10m 풍속과 -2/-1/0/+1/+2시간 시프트하며 상관을 보면 시프트=0에서 피크(r=0.727)이고 다른 모든 시프트에서 대칭적으로 하락(0.712~0.723).

## 7. SCADA 데이터 품질 `[검증됨]`

- **Vestas SCADA에 ~868개(전체 파워 판독값의 0.046%) 센서 글리치성 극단값이 있음** — 그럴듯한 최대치(~3,600kW) 대비 1e7~5×10^7 스케일의 크기, 항상 해당 행의 터빈 컬럼 딱 하나에서만 발생, 인접 타임스탬프에서 *같은* 크기의 ± 쌍으로 자주 나타남(예: 20:20에 `-11765590`, 인근 행에 `+11765837`) — 정수 오버플로우/센티널값 로깅 버그로 보이며 실제 판독값이 아님. **SCADA 기반 피처를 만들기 전에 반드시 필터링 필요**(예: `abs(power) < 5000`), 안 그러면 파워커브 피팅이 소수의 행 때문에 심하게 왜곡됨. 759/157,819행(0.48%) 영향.
- **음수 파워값은 흔하고 대체로 작음**(전체 Vestas 판독값의 16.75%가 음수, 5백분위수 ≈ -5kW, 글리치 행을 필터링하기 전에도 -100kW 미만은 434/1.89M건뿐). `[가설]`: 작은 음수값은 그럴듯함(터빈이 유휴 상태에서 요잉/제어/보조부하용 계통전력을 끌어 씀), 데이터 오류가 아닐 가능성 — 독립적으로 확인은 안 됨.
- **SCADA 파일 내에서 제조사별로 풍향 컨벤션이 다름**: Vestas `*_wd`는 `[0, 359]`(표준 컴퍼스 방위). Unison `*_wd`는 `[-180, +180]`이고 평균이 0 근처가 아니라 음수(≈-30~-64°) — **같은 컨벤션이 아님**, 재조정 없이 두 소스를 섞으면 사이트 간 방향 기반 피처가 오염됨. Open Questions 참고.
- **LDAPS의 `heightAboveGround_50_50MUmax/MUmin/MVmax/MVmin` 필드는 일관된 벡터를 이루지 않음.** `sqrt(Umax²+Vmax²)`을 단순히 "50m 최대풍속"으로 계산하면: 22.7% 행에서 이 "min" 크기가 "max" 크기를 초과함 — `Umax`와 `Vmax`가 거의 확실히 시간 내 U성분 최대값과 V성분 최대값을 *각각 독립적으로* 나타내는 극값 통계이지, 하나의 돌풍 이벤트를 나타내는 동시 벡터 성분이 아니기 때문. `(Umax, Vmax)`를 단일 벡터로 취급하면 돌풍 속도를 과대평가하게 되고, 현재 계획대로면 방법론적으로 틀림. Open Questions 참고.

## Open Questions

아래 후속 조사에 사용한 우선순위: 데이터만으로 즉시 해소 가능하고 피처엔지니어링 설계를 블로킹하는 항목(Q7, Q1, Q2, Q3)을 먼저, 맥락 참고용이거나 외부 자료로 막힌 항목(Q5, Q4, Q6)을 나중에. 각 항목은 **해소됨**, **부분 해소됨**, **미해결** 중 하나로 표시.

### 1. Unison SCADA 풍향 컨벤션 — **해소됨**

17,537개 매칭 시간에 대해 Unison 시간평균 `wd`를 LDAPS 파생 풍향(u/v → 컴퍼스 "from" 방위)과 순환상관(circular correlation)으로 3가지 가설을 검정:

| 가설 | 변환 | 순환상관 |
|---|---|---:|
| A — 이미 컴퍼스 방위, 부호만 다름 | `wd % 360` | **0.610** |
| B — 180° 어긋남 | `(wd + 180) % 360` | -0.610 |
| D — 좌우반전 | `(-wd) % 360` | -0.210 |
| 중앙값 오프셋(A vs LDAPS 방향) | — | **-3.0°** (표준편차 48.3°) |

**가설 A가 확실히 승리**: Unison `wd`는 그냥 일반 컴퍼스 방위이고, `[0, 360)` 대신 `[-180, +180]`로 표기된 것뿐 — 상대방위/나셀 오프셋 변환 불필요. 수정은 한 줄짜리 `% 360`. 48.3°의 산포는 예상 범위임(농장평균 LDAPS 방향 vs 개별 터빈 실측 방향, 지역 지형/wake에 의한 편향) — 매핑에 반하는 증거가 아니라 오프셋이 거의 0인 상태의 노이즈일 뿐. `[검증됨]`

### 2. LDAPS 50m `MUmax/MUmin/MVmax/MVmin` 의미 — **해소됨(피처 설계 결정으로서)**

벡터로서 비일관성은 확인됨(§7) — 그 부분은 그대로 유지. 개별 성분이 단독으로도 신호를 갖는지 확인:

| 피처 | `kpx_group_1`과의 상관 |
|---|---:|
| `ws10`(기준) | 0.727 |
| `Umax` | 0.681 |
| `Umin` | 0.685 |
| `Vmax` | 0.270 |
| `Vmin` | 0.296 |
| `range_U = Umax-Umin` | -0.028 |
| `range_V = Vmax-Vmin` | -0.052 |
| `sqrt(Umax²+Vmax²)`(방법론적으로 틀린 조합) | 0.733 |

`Umax`/`Umin`만으로도 완전한 `ws10`에 거의 근접한 신호를 가짐(사이트의 지배적 풍향축이 U 중심임 — 리포트 다른 곳의 wd/방향 결과와 일치). range 항목(자연스러운 "돌풍성" 프록시)은 원시 출력 레벨과는 선형 신호가 거의 없음(~0) — 원리적으로 쓸모없는 건 아니고(나중에 분산/분위수 모델링엔 의미 있을 수 있음) 1순위 피처는 아님. 흥미롭게도 "틀린" `sqrt(Umax²+Vmax²)` 조합도 실증적으론 상관이 괜찮음(0.733) — 동시성과 무관하게 바람이 센 시간대는 max-U와 max-V가 둘 다 높아지는 경향이 있기 때문 — 하지만 §7에서 보였듯 이건 "최대 돌풍 속도"가 아니므로 그렇게 문서화/추론하면 안 됨.

**결정**: `Umax`, `Umin`, `Vmax`, `Vmin`을 가짜 벡터 크기로 합치지 않고 독립된 스칼라 4개로 모델에 투입. `range_U`/`range_V`는 낮은 우선순위. `[검증됨]`

### 3. Hub-height(허브높이) 외삽 방법 — **해소됨: 생략**

GFS 바람으로 power-law(`ws117 = ws10·(117/10)^α`, α 중앙값=0.169 — 이런 지형이라면 물리적으로 타당)와 log-law(GFS 80/100m 페어에서 풀어낸 조도길이 `z0`, 중앙값 `z0≈0.57m`, 이걸 117m로 투영) 둘 다 피팅해서 원시 레벨 대비 `kpx_group_1` 상관을 비교:

| 피처 | 상관 |
|---|---:|
| `ws10` | 0.585 |
| `ws80` | 0.597 |
| `ws100` | 0.601 |
| `ws117`(power-law 외삽) | 0.601 |
| `ws117`(log-law 외삽) | 0.601 |

두 외삽 방법 모두 소수점 3자리까지 순수 `ws100`과 **동일**함 — 117m는 100m와 충분히 가깝고, 그 구간에서의 시어(shear)도 log-height에 대해 충분히 선형적이어서, 명시적 외삽이 원시 100m 필드를 그대로 쓰는 것보다 측정 가능한 이득을 주지 않음. `z0≈0.57m` 자체도 능선/해안 사이트치고는 다소 높음 — 가까운 두 레벨(80/100m)만으로 시어를 피팅한 데서 온 아티팩트일 가능성이 높고, 신뢰할 만한 독립적 조도길이 추정치는 아님 — 다른 곳에 재사용하지 말 것.

**결정**: `ws80`/`ws100`(그리고 `ws10`)을 그대로 피처로 사용; M3에서 물리적 hub-height 외삽 단계는 만들지 않음 — 여기서 입증된 이득 없이 복잡도만 추가함. 나중에 ablation에서 모델이 필요로 함을 보여줄 때만 재검토.

### 4. Curtailment(출력제한) / 운영제약 신호 — **미해결**

100% 근처 plateau 없음(§3), 제공된 어떤 파일에도 curtailment 플래그 없음, 대조할 독립적인 curtailment 기록도 없음. 이 데이터셋만으로는 해소 불가능. curtailment가 드물거나 정격이하 풍속 이벤트와 구분 불가능하다고 가정하고 진행; 학습된 모델의 잔차에서 curtailment로 설명되는 체계적 패턴(예: 고풍속에서 일관된 과대예측)이 보일 때만 재검토.

### 5. 공유되는 짧은 라벨 결측 구간의 원인 — **부분 해소됨(가설 하나 기각)**

플래그된 3개 갭 전후의 LDAPS 풍속 확인:

| 갭 | 갭 전후 평균 ws10 | 최대 ws10 |
|---|---:|---:|
| 2024-07-11 12:00–14:00 | 1.93 m/s | 3.05 m/s |
| 2024-07-30 12:00–14:00 | 7.06 m/s | 9.44 m/s |
| 2022-10-24 09:00 → 2022-10-27 18:00 (82시간) | 3.26 m/s | 6.88 m/s |

세 구간 모두 고풍속/폭풍 조건과 겹치지 않음(전부 cut-out 영역과는 거리가 멂). **이걸로 "악천후에 의한 텔레메트리/안전 정지" 원인 가설은 기각됨.** 날씨와 무관한 예정된 정산시스템/보고 갭 쪽이 더 그럴듯함. 정확히 어느 쪽인지는 여전히 미확정 — 모델링을 막진 않음(이미 권장한 대로 해당 시간을 제외하면 됨), 다만 폭풍 가설만은 미검토 상태가 아니라 확실히 기각된 상태로 바뀜.

### 6. FICR 정산단가 구간표 — **미해결, 외부 자료 필요**

PLAN.md §1.2/§4.4와 동일하게 미해결 — 평가페이지를 브라우저로 열거나 스크린샷 확인이 필요함; 제공된 CSV만으로는 해소 불가능. 여전히 보류 상태; EDA/피처엔지니어링 작업은 막지 않고, 최종 loss-shaping/보정 결정에만 필요함.

### 7. LDAPS 그리드 공간 축소 방법 — **부분 해소됨(범위만 확정, 결정은 안 함)**

그룹별 최근접 LDAPS 그리드 기준으로 3가지 축소 방법을 프록시 테스트(전체 CV가 아니라 타겟과의 상관):

| 그룹 | mean-of-16 | nearest-1-grid | IDW |
|---|---:|---:|---:|
| kpx_group_1 | **0.730** | 0.615 | 0.714 |
| kpx_group_2 | 0.740 | **0.757** | 0.751 |
| kpx_group_3 | 0.734 | **0.779** | 0.754 |

어느 한 방법이 일괄적으로 우세하지 않음. 그룹1은 mean-of-16이 명확히 우세(단일 최근접 그리드는 더 노이지/나쁨) — 그룹1의 중심좌표가 그리드 클러스터 안쪽 깊숙이 있어서(Δ≈0.009°, §2) 평균이 노이즈를 줄여줌. 그룹2와 3은 nearest-grid가 mean보다 근소하게 나음 — 이 그룹들의 중심좌표가 특정 그리드에 더 가까이 있어서, 단일 인접 그리드가 더 대표성이 있고 더 먼 그리드에 덜 희석됨. 차이는 실재하지만 크진 않고(0.73 vs 0.78) 이건 상관 프록시일 뿐 CV 결과가 아님 — 원시 상관이 괜찮아 보여도 단일 노이지 그리드는 전체 모델에서 여전히 과적합될 수 있음.

**결정**: 방법 하나를 하드코딩하지 않음. `src/features.py`에 세 방법(mean, nearest, IDW) 모두를 후보 피처셋으로 만들어두고, 이 EDA 패스만으로 결정하는 대신 Milestone M3의 CV가 그룹별로(또는 피처셋별로) 고르게 함 — PLAN.md §4.5의 원래 의도와 일치하되, 이제는 막연한 미지수가 아니라 구체적인 수치 범위로 기대치를 정할 수 있음.
